# Parameter Documentation

Canonical descriptions for the parameters defined in
`parameter-definitions.json`. Each `## <id>` section is the help text
shown in the installation UI (rendered as Markdown via the help-icon
expand panel).

The validator enforces *exactly one* description source per parameter:
a section here OR a JSON `description` field — not both, not neither.
Use this file for rich, multi-paragraph help; keep JSON `description`
for parameters where a single tooltip-line is enough.

The "### Used by" sub-section is auto-generated by
`scripts/migration/06-seed-md.mjs`. Visible bullets list addons and
applications only; templates and frameworks live in the HTML comment
block (visible in the source, hidden in rendered Markdown). Edit
anything outside the `<!-- used-by:start -->` / `<!-- used-by:end -->`
markers freely.

A "_registry-only_" Used-by line indicates that the parameter is
defined and referenced by some template, but that template is not
wired up from any addon / application's installation phases. This
is usually one of two things:

- **a runtime-only mechanism** — refresh templates (`refresh/*.json`)
  and test fixtures (`test-python-shebang.json`) are reached by their
  own runtime APIs, not by the install pipeline. Expected.
- **a missing JSON wiring (bug)** — the template is dead code from the
  installation perspective. See <https://github.com/proxvex/proxvex/issues/216>
  for the catalogue of known cases.

## ACCESS_TOKEN

Fine-grained GitHub PAT with `Administration: Read and write` on the repository. The runner uses it **once** at first boot to exchange the PAT for a runner-registration token; the PAT itself is not persisted in the container.

### When needed
Required when first setting up a github-runner container. Once the runner is registered, the PAT can be removed from the container as a hardening step — the runner then uses its own registration token.

### Where the value comes from
The user creates a fine-grained PAT in GitHub Account → Settings → Developer settings → Personal access tokens, scoped to the target repository with `Administration: Read and write`.

### Example
- `github_pat_11ABCDEFG_…` (visible only once at creation — copy immediately)

### Used by

<!-- used-by:start -->
- github-runner
<!-- used-by:end -->

## acme_needs_ca_cert

Set in `application.json` to declare that the application requires the ACME-issued CA chain certificate (e.g. for TLS termination inside the container, where the app verifies its own peer chain). When `true`, the addon writes `chain.pem` alongside the server cert.

### When needed
Only for applications that explicitly consume the CA chain. Most apps that just terminate TLS via the SSL addon do not need it — they get `fullchain.pem` instead.

### Where the value comes from
Set as a property in the application's `application.json` (`{ "id": "acme_needs_ca_cert", "value": "true" }`).

### Used by

<!-- used-by:start -->
- addon-acme
<!-- used-by:end -->
